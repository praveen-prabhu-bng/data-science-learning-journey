Introduction to Tree Traversal¶

We continue on the topic of trees, and in this section we will learn how to traverse one tree. Traversal is defined as a process of visiting each node only one time, and that's basically the complete definition. We have to make sure we don't miss any nodes, and we only have to visit them once Remember, unlike an array, a tree is not indexed. We can't instantly access nodes with one operation, so we need to start at the root, because that's the only entry point to our tree. And then use the left and right properties of our nodes to visit them. Okay, so after we visited The Root, we have a huge range of possibilities of how we can move from one node to another. We could visit the tree from top to bottom, level by level. We could go all the way down on a branch before moving to the next one. We could start from the bottom and then move to the top. Or any other hybrid combination. In theory, there are as many options to traverse a tree as there are permutations of nodes, but that doesn't mean that they're all useful, or even if they make sense

When talking about tree traversals, there are two main ways of doing it. Breadth first and depth first. And the depth first method has in turn three variations

  • in order,
  • pre order and
  • post order

Breadth First Traversal¶

We begin with what's probably the most intuitive method of traversing a tree, and that is breadth first. As the name suggests, when doing this type of traversing, we will work horizontally from one side of the tree to the other. This means that we prioritize visiting all the nodes of each level before moving down to the next one

We start at the root (29), since that's the entry point in the tree, and it's the only node at that level and we visit it. After that we move down to its children and we visit both of them (15 & 44). So again, we don't visit the nodes all the way down on one branch and then come back up. We want to make sure we visit everything on the current branch before moving down to the next one. There is, however, no rule when talking about the horizontal direction. This means that we can visit nodes either from left to right

Code Implementation

Let's say that we start at the root and we visit it. After that, we want to move to the next level. And we do that by using the left and right properties of the root to access its children. But how do we move to the next level after that? How do we visit the children of 15?Do we start at the root and access its left child, and then go to nine and 22?This approach is possible in theory, but quickly becomes more and more impractical as we go down the tree.

The solution is to use some sort of container to store nodes as we move from one level to the next. To be able to impose the breadth first approach of visiting the nodes, that container will have to be a queue. And we start by adding the route to the queue. The next steps will then be done in a loop for as long as the queue is not empty. We pop from the queue one time and we save that element in a variable. We then mark it as visited in our examples. We do that by just adding them to a list of visited nodes. After that we add both its children to the queue.

Analogy to highlight Differences Between List and deque

Think of a list like a row of lockers in a hallway: you can jump to locker #57 instantly, but if you want to add one at the front, you have to shift all lockers down.

A deque is more like a train with carriages: you can easily attach or detach carriages at either end, but finding the 57th carriage means walking through them one by one.

A deque (short for double-ended queue) is a linear data structure that allows insertion and deletion from both ends—front and rear. It can function as both a queue and a stack, making it more versatile than standard queues or stacks. However, BST (Binary Search Tree) is a hierarchical data structure used for efficient searching, insertion, and deletion operations. A deque is not inherently a part of a BST, but it can be used in BST-related algorithms, such as level-order traversal or iterative depth-first search, where elements are processed from both ends efficiently. Think of deque as just a holder—a queue used during traversal. It temporarily stores references to nodes that already exist in your tree so they can be visited in the correct order.

# We are not going to create a new tree but use an already avaiable tree. This function will only be about Tree Traversal          
    def breadth_first_traversal(self):
        """ Performs a level-order traversal : Nodes are visited level by level from top to bottom, left to right."""
        if not self.root:
            raise Exception("Tree is Empty")
        queue = deque()
        '''creating an empty double-ended queue (a deque)from the collections module.It allows fast appends & pops from both the front and the back. 
        It's much more efficient than using a list when doing repeated insertions/removals from the front(like in BFS),we are initializing the queue 
        that will be used in the Breadth-First Search (BFS) traversal of the tree.It's where nodes are temporarily stored (for processing) until it's 
        their turn to be visited '''
        queue.append(self.root) # - Adds the root node of the tree to the queue.
        visited = [] # Initializes an empty list to collect the values of nodes as they are visited. Not the nodes themselves, just their data.
        while queue: # - Loops as long as there are nodes in the queue.
# popleft():Removes & returns the node at the front of the queue.This allows for a (FIFO) traversal order,perfect for level-order scanning of the tree.
# we pop from the queue(from front which is to the left) and save it in visited list for printing output            
            visited_node = queue.popleft() # This is the current node being visited and visited_node is reset everytime the while loop runs. 
# since we appended the self.root as the first element in the queue, its the first to get poped left.            
            visited.append(visited_node.data) # The attribute you defined to hold the node's value is called data, not values
# Key diff betn "visited_node" & "visited". Append adds new elements to "visited" while values are overwritten in "visited_node"every time the loop runs
# if you are not using ".append" to a variable, the value will be overwritten every time.
            if visited_node.left:# since visited_node is reset everytime,we capture left & right child(whichever present) of a new parent node everytime
                queue.append(visited_node.left)# while visited.append is for record keeping, queue.append is for the purpose of traversal control 
            if visited_node.right:# if visited_node has left or right nodes, it will be appended into queue so we can popleft and further append it 
                queue.append(visited_node.right)  # into visited, this loop continues till we reach the leaf
        return visited

Quick analogy on visited_node vs visited

Think of it like this:

  • Assignment () → Replace the text on a whiteboard. Each loop erases the old text and writes new text.
  • Append () → Add a new sticky note to a wall. Each loop adds another note, so you keep the whole history.

the deque only contains one element: the root node of the tree. It does not magically know about the entire tree. It’s just a container, initially holding the starting point.

How do we “read” the tree then? The traversal happens dynamically:

  • Start with root in the queue.
  • Pop the front of the queue (popleft()).
  • Process that node (e.g., record it in visited).
  • Enqueue its children (left and right, if they exist).
  • Repeat until the queue is empty.

So the tree is not preloaded into the deque. Instead, the deque grows and shrinks as you discover

Analogy

Think of the deque as a to‑do list:

  • At first, your to‑do list has just one task: “visit the root.”
  • When you finish that task, you add new tasks: “visit left child” and “visit right child.”
  • The deque ensures you always handle tasks in FIFO order, which gives you level‑by‑level traversal.

Time And Space Complexity¶

what about the time complexity of the breadth first traversal? What do you think that would be?

What we need to be looking at in order to answer that is the while loop. Because this is not a for loop where we could instantly see over how many items we're iterating. Assessing the time performance might seem a bit more difficult, but in reality it's not. One hint we have inside the loop is the number of nodes we're visiting with each iteration.And as we can see, that number is one. So we're visiting one node with each loop.This means that we'll be doing as many loops as there are nodes in the tree.So for a tree of n nodes we would have an O(n) complexity.

Let's now check the space complexity.In our method.We're creating two additional containers where we store some data a queue and the visited list.And visited list is there because that's our definition of visiting a node. So technically we should analyze the performance of this algorithm without it, since in any other circumstancevisiting a node might not use extra memory. So if we ignore the visited list and we focus only on the queue, what would the memory usage be?

To answer that, we need to think about how we use the queue to store the nodes.It's true that we're adding all the nodes to the queue, but they're not all of them in there at the same time. We're freeing some memory when we pop from the queue and we visit the node. Right now we don't have that many nodes stored in memory, but as we keep moving down, the queue will start holding more and more of them.Basically, the queue will occupy the most amount of memory when we reach the widest part of the tree,which is typically at the bottom of the tree.So even though we almost never store the entire tree in the queue, the amount of nodes we have in memory at any time will grow proportionally to the size of the tree.So the space complexity is also O(n)

Time : O (n)¶
Space : O (n)¶

Depth First Traversal (DFT) Pre Order Iterative¶

If Breadth first requires us to visit the tree level by level and move down only after finishing the current level, depth first is the opposite of that. In this type of traversing, we proceed as far down as we can on one branch, and when we reach its end, we go back up to the nearest crossroad and then go all the way down on the next branch, and so on. And we can start this process from left to right like we did here. Or from right to left. Both of them are valid depth-first strategies. As long as we prioritize going down the branch before doing its side to side. Although this method gives us a clear rule of how to move from one node to another, it doesn't specify when we visit the nodes

Do we visit them as we move down the tree? Do we first go all the way down and start visiting when coming back up? These types of questions have prompted a few variations of the same traversing method to be developed, namely the preorder, the in-order, and the post-order traversals. They all have a depth first approach, but they all visit the node at different times during the process.

Putting this type of logic in code would seem a pretty daunting task, but it's actually surprisingly easy if we use another data structure we already learned... THE STACK.

Queue uses FIFO which works fine with Breadth First Traversal (Add to Queue-> Popleft -> find children -> Add to queue)

while Stack which uses LIFO is best fit for Depth First Traversal Tree.

When it comes to the Code, the entire method is extremely similar to the breadth first traversal. The only difference is that we're using a stack instead of a queue, and we therefore prioritize visiting the last node that was added to our stack. Everything else is pretty much the same

We start at the root and before moving to any of its children, we mark it as visited. And then we need to choose one path to go to, either to the left or to the right. For the iterative version, we'll go to the right side first, and for the recursive version we'll go to the left one.

So after the root(29), we go to its right child(44) and we mark it as visited. After that we don't move to 15 but continue downwards to 49.We visit it and then to 58. Since we now reached the end of the branch, we go up one step back to 49 and check if it has any left child. If it does, we visit it and now we check for the right child of 46. But since it doesn't have one, we go to 45 and we visit that instead. Now that we visited all the nodes that stem out of 49, we go again one step up to the next branching, which is at 44.We then visit 40, and since it doesn't have a right child, we check its left one. We visit 35 and then check for the right child. This time we have one 39, so we mark it as visited. And since we reached the end of this branch, we go back to 35 and check for the left child31. We visit that as well and we're done with the right side of the tree. Going back one more time to the next branching would mean going back to the root29. Visiting the left child 15and then repeating the same process for all the remaining nodes.

Depth First Traversal (DFT) Pre Order Recursive¶

When it comes to any of the depth first traversal algorithms, you'll almost always see them implemented using recursion, and there's a good reason for that. They require way less code, and to some people they're more intuitive than their iterative counterparts.

We have discussed both versions in this section, just to give you an idea of how two implementations of the same algorithm might differ. Just to be clear, this is still the preorder depth first traversal. We're still traversing the tree the same way we did in the previous section, but this time we'll use recursion instead of iteration.

def dft_pre_order_recursive(self):
    if not self.root:
        raise Exception("Tree is empty")
    visited = []

    def _traverse(node):
        if node:
            visited.append(node.data)
            _traverse(node.left)
            _traverse(node.right)
        return

    _traverse(self.root)
    return visited

In this Recursion function , when the node has children , it dives deeper, but the Parent loop doesn’t close till the children loop has finished. Each new _traverse(...) call goes on top of the stack as highlighted above(just like stacking plates). When a call finishes, it's removed from the top (unstacked).

Its starts with Root(29), 29 is appended to "Visited", then 29.left (15) function is activated ..now the baton is passed to 15 , but 29 loop is still open, since 29.right function is waiting for 29.left function to finish. Node 15 follows the same pattern and passes the baton to 9 (15.left) ..while 15.right is waiting and so is 29.right…this goes on till we reach the last node/leaf and when we get node.left =None (2.left=None)as response, the function returns immediately and pops off. Now we are still within _traverse(2) function, _traverse(2.left) has returned None, we now activate _traverse(2.right), which also returns None. This completes _traverse(2) loop and the control returns to previous/parent branch _traverse(5). Now _traverse(5) will execute its second function _traverse(5.right) which results in _traverse(8), when this loop also closes, the baton is passed back to _traverse(5) and when all functions with _traverse(5) are completed, the baton is passed back to parent node _traverse(9). This continues till it reaches root(29) and now _traverse(29) will execute its pending function _traverse(20.right)

Think of each _traverse() call like jumping into a tunnel. When the node has children, you dive deeper. But when you hit None, you pop out and return to the previous junction, continuing in a different direction This back-and-forth behavior is what gives recursion its power: it dives down a branch as far as it can go, and as soon as it hits a dead

Both the recursive and iterative versions are called pre-order because they follow the same visiting order of nodes in a binary tree:

Root → Left → Right

That’s the defining trait of pre-order traversal, regardless of how you implement it.

🧠 So why the name “pre-order”? The term comes from the position of the root node in the traversal sequence:

  • In pre-order, the root is visited before its children.
  • In in-order, the root is visited between its children (Left → Root → Right).
  • In post-order, the root is visited after its children (Left → Right → Root).

So the “pre” in pre-order means the root comes first—not that it’s recursive or iterative.

As long as the traversal visits nodes in Root → Left → Right order, it’s pre-order—no matter how you get there.

Another side note, you might have noticed that we now went to the left side of the tree as opposed to the previous section where we started with the right one. And that's just due to the order of which we called the traverse functions (node.left & then node.right). We would achieve the same order as the iterative version if we were to traverse the right child first and then the left one. This is just a detail that matters only if you specifically want to prioritize one side over the other. It doesn't change the fact that we're still traversing preorder depth first

Time And Space Complexity¶

Regarding complexities and like all the other versions, they will be O(n) for both time and space. Please note that we're still using memory, even if we don't have a dedicated stack. We're basically using the systems call stack to achieve the same result we did in the iterative version. And if you pay attention, you'll see that the call stack will only grow with as many frames as the number of nodes in the deepest branch of the tree, because we're constantly popping off functions as they finish executing.

def dft_in_order_iterative(self): # (Left -> Root -> Right)
    if not self.root:
        raise Exception("Tree is empty")
    current_node = self.root
    visited = []
    stack = []
    while stack or current_node: # while loop will never start if stack is empty, hence also add current_node
        ''' This section lets you reach to the leaf. Will be true till it reaches the Left leaf of the tree '''
        if current_node: 
            stack.append(current_node)
            current_node = current_node.left # to reach the left most end or leaf of the structure
        else: # if we reach the end of the branch
            ''' This section will help you traverse from left lowest nodes to Root and then cover Right side nodes '''
            visited_node = stack.pop()
            visited.append(visited_node)# Now we start from the left most leaf of the tree & start traversing and capturing the nodes(left-Root-Right)
            if not visited_node.right: # if there is no right child of the leaf then use continue to go back to While loop

If we move this line down to visit the node after we traverse the left child, we just changed our algorithm from pre order to in order. And this is not a joke. It's that easy. We're now prioritizing traversing to the left over visiting the current node. You can see why the recursive versions of these methods are preferred over the iterative ones. def dft_pre_order_recursive(self):

In the context of binary trees—and specifically in-order traversal—what someone likely meant is this: The standard in-order traversal visits nodes in the order:

  1. Left child
  2. Root
  3. Right child

So if your tree stores values in a way that resembles a Binary Search Tree (BST), this left-root-right traversal will print the values in ascending order.

  1. Right child
  2. Root
  3. Left child

you get a reverse in-order traversal, which means the nodes are visited in descending order. Here’s what that might look like in code:

Depth First Traversal (DFT) Post Order Iterative¶

Post-order is in a way resembling in-order in the sense that we'll again traverse down a branch, but only visit the nodes on the way back up. The difference is again in the order we make the visits.

Previously (DFT In order Iterative) we went left side(2) parent(5), then right side(8). With post order. We will do the left side(2), then the right side(8) and after that the parent(5)

So in our case here we would visit 2, then its sibling 8, and only after that go to their parent 5. And now we go one level up and work on the sub tree with 9 as the parent. Since we did its left side, we must now visit its right one, so we would traverse again downwards until we can't anymore and then go again left right parent.10 does not have a left child, so there's nothing to visit, but it has a right one. So we visit 12 and then we visit the parent. With both sides of 9 visited. We can now visit 9 itself. After that we go one level up again. We just did the left side of 15 so we moved to the right one. We again try to go as much to the left and down (21). We end up at the mini tree with 19 as the parent. We visit 21, we visit 19, then move up. Here we have the right side of 22 which is only 27. After we visit those, we can finally visit 15. So that's how Post-order looks like

When coders design an algorithm like iterative post-order traversal, what they’re really doing is reverse-engineering the behavior of a recursive call stack—which is invisible but incredibly powerful. They ask themselves: “If recursion works by holding a place on the system call stack until left and right are processed, how do I replicate that logic explicitly using my own stack?”

To make that happen, they needed a clever strategy to:

  • Track where they came from → using a stack.
  • Know what’s already been visited → using previous.
  • Avoid revisiting right children → with that subtle condition you now understand.

It’s like building a map with signs that say “I’ve already explored this path, don’t go there again.” That level of precision takes both logical thinking and creative insight.

Coders often sketch it out by hand, simulate tiny trees, debug dozens of edge cases, and refine until it behaves like nature intended. That mental model—the one you just built from tracing node 2 to 29—is the same one they used to create it.

Depth First Traversal (DFT) Post Order Recursive¶

In the preorder version, we visit the node prior to traversing to the left and right . The in order does the visiting in between traversing the left and right children (below). So naturally Post order will do the visiting after the traversing part.

def _traverse(node): # Pre Order = Root → Left → Right. Use case: Often used to copy or serialize a tree,because you record the root before its children.
    """ DFT Pre Order Recursive """
    if node:
        visited.append(node.data) # code # 1
        _traverse(node.left)      # code # 2
        _traverse(node.right)     # code # 3
    return

def _traverse(node): # In Order = Left → Root → Right. Use case: For Binary Search Trees (BSTs), this traversal gives you the nodes in sorted order
    """ DFT In Order Recursive """
    if node:
        _traverse(node.left)        # code # 2 swapped with code # 1
        visited.append(node.data)   # code # 1 swapped with code # 2
        _traverse(node.right)       # code # 3 remains as it is
    return

def _traverse(node): # Post Order = Left → Right → Root.
    """ DFT Post Order Recursive """
    if node:
        _traverse(node.left)
        _traverse(node.right)
        visited.append(node.data)
    return

def dft_post_order_recursive(self):
    if not self.root:
        raise Exception("Tree is Empty")
    visited = []

    def _traverse(node):
        if node:
            _traverse(node.left)
            _traverse(node.right)
            visited.append(node.data)
        return

    _traverse(self.root)
    return visited

Summary¶

So we just learned two main approaches of traversing a tree 1.breadth first and 2.depth first. And they ultimately do the same thing. They visit all the nodes in a tree. So assuming we don't have any other reason to do one over the other, let's think which one would be better in certain scenarios in terms of performance. It's true that they all have an O(n) complexity, but there are some cases where one would be a better choice

This is a perfectly balanced tree, one which grows exponentially the deeper it goes. In a breadth first traversal, we would go level by level and use a queue to store the nodes we'll visit next. And we saw that the queue will contain, at some point all the nodes that are found at every level.

So at the top it will contain only the root. Then the two nodes at the second level, then these four and so on. And when it reaches the bottom level, the queue will have all the leaf nodes in memory at the same time. In this type of tree, that number would be about half the total number of nodes. For large trees, or if we're short on available memory, that could be an issue.

With depth first though, where storing the nodes that go from the top of a branch to the bottom. And in this type of tree, you can see that the number is considerably smaller than what we had before. So a depth first approach would be a better choice in this case

But what about this tree below ? Here we have one which is deeper, but much narrower at its widest. We only have three nodes, so a breadth first traversal will only keep at most three nodes in memory at any point. If we were to traverse it depth first, the depth of the branch will dictate the number of nodes we have to add to the stack. And you can see here that there's a big difference between the two methods.

The point is, if visiting every node is the sole purpose, regardless in which order, and you need to be conscious about memory utilization, choosing the best method depending on the structure of your tree would be a good idea.

Let's also have another look at the three depth first variations and when we would want to use one over the others. This would be the order we would get if we were to traverse the tree. Preorder .This method is typically used when we want to linearize a tree, that is, to represent it as a list.

This will help us if we want to make a copy of the tree, because if we were to iterate over the list of visited nodes and insert them in a new binary search tree, the nodes will be inserted in the same order as before.

So the preorder method preserves the structure of the tree if we want to use it at a later moment. In order would give us this result if the tree were traversing is a binary search tree. The list we get back would actually be the ascending order of our elements. Traversing from the right side will of course, give us the descending order. This might be useful to find out the structure of the tree.

For example, to see how many elements you have that are less than or greater than the root. As we already learned, post order would visit the leaves first before moving up to the parent node. This in and of itself can be a reason to choose this type of method, but another practical use of post order is to delete a tree from the leaves up. So it's all situational, as is everything with programming. We choose our solutions based on the problem we're trying to solve

Tree Traversal Use Cases (BFT vs DFT)¶

📌 Breadth‑First Traversal (BFT)¶

Definition: Explore nodes level‑by‑level (horizontal expansion).

Real‑Life Use Cases¶
  • Shortest Path in Unweighted Graphs
    GPS navigation, routing protocols, or network packet delivery.
  • Social Network Analysis
    Finding friends‑of‑friends or community detection.
  • Web Crawling
    Search engines crawl pages breadth‑wise to discover all links at a given depth before moving deeper.
  • Garbage Collection (Memory Management)
    Mark‑and‑sweep algorithms traverse object references breadth‑wise.
  • Disaster Spread Simulation
    Modeling wildfire spread or disease transmission, where expansion happens in waves.
📌 Depth‑First Traversal (DFT)¶

Definition: Explore nodes branch‑by‑branch (vertical expansion).
Types: Pre‑Order, In‑Order, Post‑Order.

Real‑Life Use Cases¶
  • Expression Tree Evaluation (Post‑Order)
    Compilers evaluate arithmetic expressions bottom‑up.
  • Route Finding in Maps (DFS)
    Explore all possible paths deeply before backtracking.
  • Topological Sorting / Dependency Resolution
    Build systems (make) or task schedulers ensure dependencies are resolved before parent tasks.
  • File System Traversal
    DFS is used to list or delete directories (children before parent).
  • Game AI (Minimax)
    DFS explores all possible moves down to terminal states before evaluating.
📊 Comparison Table¶
Traversal Exploration Style Typical Use Cases Key Advantage
BFT Level‑by‑level (queue) Shortest path, social networks, web crawling, garbage collection, spread simulation Guarantees shortest path in unweighted graphs
DFT Branch‑by‑branch (stack/recursion) Expression evaluation, route finding, dependency resolution, file systems, game AI Memory efficient, explores all paths
✅ Key Insight¶
  • Use BFT when you need shortest paths or level‑wise exploration.
  • Use DFT when you need complete exploration, dependency resolution, or bottom‑up evaluation.
