Trees Intro¶

In the previous section, we briefly mentioned that arrays and linked lists belong to this group called Linear Data Structures. This means that they're all limited to only one dimension. They can't represent any kind of hierarchy between their elements. To overcome this issue, new data structures were created. Ones that can be organized on more than one level and can reflect some sort of hierarchy. One of these structures is the tree. Just like linked lists, trees are also built using node objects, but this time they follow different rules

A tree contains one node that is at the top level, which branches out into other nodes, and those nodes can in turn branch out into different nodes, and so on. Notice that the nodes are now placed on different levels, and because of how they are connected, we can't go from one to the next in a linear fashion like we did with linked Lists

Let's define some key terms we're going to use when we talk about trees

  • Nodes :We already mentioned nodes. They're the building blocks of trees. They share the same name with the objects we use to create linked lists, and they also carry a value and some other properties
  • Edges: Edges also called links and sometimes arcs, are the connections between nodes. You can think of them as being similar to the next or previous pointers we had with linked lists. They are the things that keep the nodes connected to the tree.
  • Parent : Parents are nodes which have connections to other nodes located in the next level below them in the tree hierarchy. In our example, 5 is a parent for both 13 and 8. And 8 is a parent for 17 and 4, and so on

One thing to note here is that we can consider a node being a parent to another node, only if it has a direct link to it. So for instance, we can't say that 8 is a parent to 25 because they don't have a direct connection between them.

  • Ancestor : 8 will be an ancestor to 25, and so would 5 if we were to go from 25 to its parent and then keep going on the hierarchy chain, all the nodes until the top level node can be considered ancestors to 25
  • Child : A node that has a parent is called a child. In our example, 22,1 and 38 are all children of 13.The parent child relation we just discussed in the previous bullet point is still valid. We can't say that 1 is a child to 5. For example, 1 is the child of 13
  • Descendant : If we wanted to describe 1 in relation to 5, we would say that 1 is a descendant of 5
  • Sibling : These are nodes that have the same parent. The node 22, 1 and 38 are also siblings. They're all children of 13.
  • Root : The top most node is called the root. The root is the only node that doesn't have a parent and can only have children. A tree can only have one root. In our tree that is the 5 node
  • Leaf : are the ones that do not have any children. They're the end of the line for all the branches of our tree. Leaves are not limited in number. A tree can have as many as possible. The leaves of our tree are 22, 1, 38, 17 and 25.Any node that is neither the root nor a leaf is usually called an inner node or internal node

Let's talk about some rules that we should follow to be considered a valid tree.

  • We already established that a tree can have only one root, although there are some implementations that can have multiple nodes at the first level of the tree. From a mathematical standpoint, a tree can have only one root
  • Another rule is that a node can have any number of children from none, in which case there leaves toas many as we want. Some implementations, however, limit that number to either 2 or 3 children
  • Every node needs to have only one parent. The exception to that is of course the root, which is the only node without a parent. But if you look at this example, you won't see any nodes that have two parents. Something like this(refer below snapshot), for instance, with 25 having both 17 and 4 as parents is not allowed. This is not a tree any more

  • The last rule would be that nodes on the same level should not be connected. For example, 17 and 4.If they had a link between them, it would make this a different data structure altogether.

Introduction to Binary Search Trees¶

why binary? The word binary has something to do with the number two in the context of a tree.

  1. Binary means that any node in the tree can have at most two children. They can have no children, in which case they are leaves. They can have one child or two, but never more than that. This tree we see in this image below is a binary tree. There is no node in this tree that has more than two children.

  2. The second rule of a binary search tree is that all the values to the left of a node have to be smaller than any of the values to its right.

  3. Duplicates are typically not accepted in a binary search tree. Having duplicates will greatly increase the complexity of the tree, and it's generally advised that a binary search tree should only contain unique values.

Another aspect to think about is what type of data we store. In our example there were numbers, but our tree can technically work with any values that can be compared, like letters, for example. Our binary search tree could have the letter L as the root, and then distribute all the letters that come before L to its left and all that come after it to the right. Keeping in mind that all the subtrees should also follow this rule

Because of how the values are stored in a binary search tree. Operations like insertion, searching, and deletion can be very efficient. This is what the term search in the binary search tree stands for. It's pretty much a statement of this data structures purpose

If we were to do a short recap, we can say that binary search trees are a hierarchical tree structure where each node can have at most two children. The left child is less than its parent and the right child is greater than it.

Insert¶

The first method we'll learn is called insert, which will be responsible for adding new nodes to our tree. And the first case we can consider is an empty tree. When a tree is empty, the root property will be none.

  1. So when we add the new node, we'll just make it the root.
  2. If a tree is not empty, we need to compare the new value to the root node and remember the rule.
  3. If it's smaller than the root, it goes to its left. And if it's bigger then, goes to its right

So a more detailed description of the process will be like this.

  1. We first need to take the value we pass as a parameter and create a new node instance with it.
  2. We then check the root of the tree. If it's none, we just make our new node the root and we end the execution.
  3. If the tree is not empty, we want to start at the root with a variable, and then write a while loop that runs only if the value we want to add is different from the current node's value.
  4. Inside the loop, we compare the value we want to add to the current node.
  5. If it's less than it, we check if the left child of that current node is none.
  6. If it is, we place the new node there.
  7. If not, we update the current node to be its left child node.
  8. If it's greater than the current node, we check if the right child of the current node is none, and
  9. if it is, the logic is the same. Place the node there and
  10. if not, update the current node to be its right child.

So that's the algorithm for adding a new node to our tree.

def insert(self, value): # Defines a method insert() to add a new node to the BST(Binary Search Tree)
    new_node = value # When we use Node(value), we are calling the constructor of the Node class. It initializes a new node with the given value
    if not self.root : #  If tree is empty 
        self.root = new_node # It assigns new_node as first node(root). This handles the case when inserting into an empty BST.This is fixed with first node value and will not change . 
        return self # After assigning Root value, exit from if loop and back to insert(value) command for next input
    current_node = self.root # current_node is used as a temporary reference to traverse the tree, If we directly modify self.root during traversal, we might unintentionally change the root of the tree
# Also it needs to be refreshed/reset, to retain the root value (outside while Loop) and while inside the while loop, it will be updated with left & right nodes based on availability in each levels/layers of left & right    
    while value != current_node.value: # a) Loops through the BST until it finds an empty spot to insert the new node. b) Ensures no duplicate values are inserted.
        if value < current_node.value: # If the new value is smaller than the current node, we move left. This follows BST rules (smaller values go left). Larger values goes to the right
            if not current_node.left: # Checks if the left child is empty
                current_node.left = new_node # if yes, inserts the new node @ left of the current_node
                break # when left side is updated with new node, it Breaks out of the while loop because insertion is complete and executes the return self code ending our function
# If left node is not empty (becoz its either higher/lower than new node), below code will make left node as new current node so we can compare again (thru while loop)
# and  find the appropriate place depending on whether new node is higher/lower than updated current node           
            current_node = current_node.left # current_node is updated with latest left node, so when the value is looped again (till while condition meets) it will be compared with the latest left node and not root value
# e.g. in below chart, when 7 comes along, first it will be compared with 20, since its < 20, we check if left child is empty, since it's not (10) this line is executed which results in 10 replacing 20 as current_node, 
# becoz 10 is on the left of current_node(20). This is the code which updates the current_node or goes 1 level down depending on the value inserted        
        else :
            if not current_node.right: # works same as Line 15 but for values which are higher than current_node(depending on diff levels). All values move to the right when they are higher than current_node
                current_node.right = new_node # if yes, inserts the new node @ right of the current_node
                break
            current_node = current_node.right # works same as Line 18, except for it looks for the child @ right 
    return self

Time And Space Complexity¶

Let's now talk about the complexity of our function. And this time we'll start with the space complexity. Does the memory utilization increase based on the size of the tree? It doesn't. We only create one new node to add to memory. And remember this variable reassignment we do inside the while loop does not consume more memory. We're just updating the same one over and over again so we don't keep the old values. So the complexity is O(1)

Let's now talk about the time complexity. And the reason why I left this at the end is that we usually describe it in terms of best and worst complexities, with the best being O(logn) and the worst being O(n). To better understand why that is, let's have a look at these two examples. This tree (left pic below) I have here is a binary search tree and you can probably notice that it's very symmetric .Every single node has two children except for the leaves at the bottom, and all of these leaves are on the same level. We don't have one side of the tree that goes deeper than the other.

This is what is called a perfectly balanced tree. And this type of tree has the O(logn) performance. To understand why. Let's say we want to add the number 20 to this tree. Like we just saw. We need to start at the root and compare the value of the root with the number 20.Since 20 is less than 29, we know for sure that 20 will be placed somewhere to the left of 29. So we can ignore all the nodes that are to the right of the root. So by doing just one comparison, we excluded half of the values of our tree. And the same thing happens when we compare 20 to 15.Therefore, we now ignore everything that comes to the left of 15.And we continue doing so until we find the place to add the new node. So in a tree of 15 nodes, we only need to do five comparisons to add our new number. With each iteration, we excluded half of the remaining nodes from comparison. This is a textbook based to log mathematical expression, and that's a very efficient and desirable complexity.

So what about the other one (right pic above)? This is what a tree with an O(n) complexity would look like, even though it looks more like a stick than a tree. It's technically a tree, and I hope you can guess why this is of O(n) complexity. Well, let's say we wanted to add the number 50 to this tree. We again start at the root, do one comparison, then move to the right. Compare again, move to the right once again and again until we reach the next available spot. Notice that with this type of tree we did five comparisons, but the tree has five nodes in total. If this was 1 million nodes, we would have to do 1 million comparisons, so the time would increase proportionally to the size of the tree. This is why this is O(n)complexity.

And what we see here are the extreme cases. One is perfectly balanced and the other one is basically a singly linked list. But most of the time when it comes to shape, binary search trees are somewhere in between. They might be skewed to one side or the other, or have branches that are either shallower or deeper. It all depends on the value of the root and the next nodes added afterwards

Time : Best - O (logn) | Worst - O (n)¶
Space : O (1)¶

Contains¶

We want to write a function that takes a parameter, and then looks around the tree to see if it contains that value

So if we were to describe this as pseudocode, the logic will be like this.

  1. Save the current root in a variable,
  2. Start a while loop that will run until this variable ends up being none.
  3. Inside the while loop, Check if the value you want to find is equal to the value inside the current node.
  4. If it is, return true and be done with it. If it's not equal, check if it's smaller.
  5. In this case, update the current variable to be its left child, and if it's larger, update it to be its right child.
  6. Lastly, outside the while loop, return false if you didn't find the value you were looking for.

And that would be the process of searching for a value in a tree

def contains(self, value):
    current_node = self.root # - Starts the search at the root node of the tree. self.root is assumed to be the top-most node of the tree.
    while current_node :# Loops as long as there is a node to check.If current_node becomes None,it means we’ve reached a leaf without finding the value.
        if value == current_node.value: # If the value matches the current_node value, it returns True, confirming that we found the value
            return True
        if value < current_node.value: # if the value is lesser than current node value, it switches the current to left node and the loop continues
            current_node = current_node.left # with the updated current_node
        else :
            current_node = current_node.right
    return False
  1. Why we don’t use .value in current_node = self.root
  • At this line, we’re not comparing anything yet — we’re just assigning a reference to the root node object.
  • self.root is itself a Node object (with attributes like value,left,right).
  • So current_node is meant to hold the whole node, not just the value.

Later, when we want to compare, we should use current_node.value.

That’s why .value isn’t used in the assignment. We want the node, not just its stored value.

  1. Why we don’t check if left/right exists before moving
  • The loop condition while current_node already ensures we only continue if the node exists.

Inside the loop:

  • If current_node.left or current_node.right is None, then current_node becomes None.
  • On the next iteration,while current_nodefails, and the loop exits → return False.

👉 So we don’t need an explicit check like if current_node.left is not None. The loop condition handles it naturally.

Remove¶

Remove is quite complex and has a few scenarios to cover, hence we are splitting it into four sections, each of them covering a separate case.

In this section we will describe what remove is and what we want to achieve with it. We'll also write some code of locating the nodes involved in removing a value. In the next 3 sections, we will explain the different situations we might encounter when we want to remove a node.

Just like the remove method we did for the linked lists, we want it to delete a node based on a value we provide. But the issue is that the nodes in a tree can be of more than one type. The removed node can be the root. It can be a leaf. It can be a node that has one child. Or one that has two children. These are all different cases we need to consider

But before we talk about how to remove a node, we first need to find it. Also, do we need to locate only the node we want to remove, or do we also need something else?

Let's take below example. The 5 node. What is keeping this node part of the tree? The answer to that is the link.It has to 9 its parent node.If we cut that link, 5 and 8 will be removed from the list.So when we'll search for our node to remove. We also need to keep track of its parent, just like we did when removing from a singly linked list.

Strategy for implementing Remove

Let's write the code for the first part of the algorithm, locating the node we want to remove. And the way to do it would be like this.

  1. We start with a current node variable that will initially be equal to the root of the tree.
  2. We also want a second variable to keep track of the parent of the current node.
  3. We then do a while loop and update both the parent and the current node variables until we either find the value we're looking for or until we reach the none value.
  4. If current node will be none, it means that the value we're looking for is not in the tree and we can stop the program.
  5. If we find our node, we need to check the number of children it has, and we can stop here for this section

def remove(self, value, start=None, parent=None):

Remove Node - No Children¶

Logic :

Let's consider below tree as an example and say we want to remove this node with the number 39.Recall that removing 39 is done by cutting the connection with its parent, which is 35.So how can we do that?

If what we implemented earlier is correct. After we locate the node we want to remove, we'll have two variables in memory the current node and its parent. So to remove the current node, we want to set the right child of the parent variable to be none. And how do we know it's the right child? Well, we can't know in advance. What we have to do is just check both children of the parent. If the left one is equal to the current node, we make it equal to none instead, and the same for the right child.

And there is also one special case of removing a leaf node. Can you guess it? It's the root when it's the only node in our tree. This is the only time when the root is also a leaf. In this case, our node to be removed does not have a parent, so we can't apply the same logic of checking the left and right children.

# All remove child subfunctions are using "_" e.g _remove_node.. which indicates private helper fn, which keeps these internal fns hidden & prevents
# external users from accessing it. It also helps in better & clean code mgmt , breaks down complex tasks into smaller, reusable functions
def _remove_node_no_children(self, current, parent):
    # Removing the Root, if its the only node in the tree
    if current is self.root: # this is the only type of Remove when we can check if the root itself is the leaf (no children)
        self.root = None # If the node is the root, set self.root = None (tree becomes empty).
        return self
    # Checking which side of the parent is the current node    
    if parent.left == current:
        parent.left = None # If the node is the left child of its parent, remove it by setting parent.left = None.
    else:
        parent.right = None # If the node is the right child of its parent, remove it by setting parent.right = None.
    return self

Remove Node - One Child¶

Scenario 1:

In the tree we have here. There are a few nodes with only one child. One of them is 40, which only has 35 as the left child. So what would removing 40 mean for the structure of the tree? In the previous case we discussed, we just made the parent point to none instead of our node. Here. We can't really do that because we'll lose all the nodes that branch out of 40, like 35, 31 and 39.What we'll have to do is to point 44 to the child of 40.This will both remove the desired node and keep in the tree the rest of the nodes that were on the branch.

Solution:

We already saw that when we find the node we want to remove, we also keep track of its parent. So the first part of the logic is the same as the last method. We need to check both the left and the right attributes of the parent to see to which side is our node located. But after that, things are different because now we also need to know to which side of 40 is its child located. So we can point 44 to it. So we also need to do a similar check for the current node. If the left child of 40 exists and is not none, we must point the 44 to it. If the right child of 40 is not none, we need to point the parent to that one instead

Scenario 2:

If our tree looks like this and we want to remove the root 29 but it only has one child 44, we can't apply the same logic since there is no parent.

Solution:

Well, we can do is again check both sides of the node and whichever is not equal to none 44, make its child the new root 44. And since the tree now has the route equal to 44, the old root 44 will be removed from memory. So these are the two situations for when the node we want to remove has only one child

def _remove_node_one_child(self, current, parent):
# Logic : 1. Check if the child (of Current) is on right or left (since we are looking for only 1 child)

Remove Node - Two Children¶

The most difficult scenario between the three, but also the most interesting, is when we have to remove a node that has two children. But why would that be so hard to do? Well, let's have a look.

Let's say that from this tree we want to remove 15.We saw until now that the way to remove a node was to basically make its parent point to either none or to one of its children. But what if we were to apply the same logic to our node here? To which child should we point its parent? Do we point 29 to 22?

We Could, But what happens to this other branch? It will basically be removed from the tree (refer below).

Should we make a connection from 9 to 29 then? Well, we can't, because if we did, this wouldn't be a binary tree any more.

So we need a different approach. The secret is to find a replacement for the removed node. We need to find another node that's in the tree that could replace 15 without breaking the rule of beingless than its entire right side, and greater than its left side. And we have more than one to choose from. To better understand how we can identify them, I think it's time to introduce two new terms the successor and the predecessor.

The successor is the node with the next larger value in the tree. In our example, the successor of 15 is 19.That's the next node greater than 15.The predecessor is the node with the next smaller value for 15, that will be 12.If we take another example, like 44, its successor will be 45.And the predecessor 40. But for our purpose, which is to replace the node we remove, we won't need to find both of them. Only one would be just fine

what is the logic of finding the successor?

The steps are as follows :

Example 1 - From the target node (15), we need to navigate to the right only once(22). And then if we have a left child (19), we need to keep going to the left until we can't any more. In this case, we only went one time because 19 does not have a left child. And when we reach the end, we don't go to the right anymore. We stop. Even if this is not a leaf node, it doesn't matter. We don't want to reach the end of the branch. We want to go only to the left as much as possible. And when we can't any more, we found our successor.

Example 2 - How about 9? With this one(9) we go right once(10), but then we don't have any left child to go to from 10.So we stop. And that's the successor of 9.And by the way, this formula of going right once and then all the way to the left works only if the node we want to find the successor for has a right child. If our node didn't have a right child like 27, for instance, the way we would find the successor(22) of 27 would be to go up(15) the tree until we turn right for the first time and then stop and we end up with 29 being the successor of 27 which is correct. But we don't have to worry about this case, because the node we will remove will always have a right child. Keep in mind, we're now handling the case where our node has two children. So one more time to find the successor of a node that has a right child, like 29, for instance. We need to navigate only once to the right(44), and then go to the left (40) and left only (35) until you can't anymore (31). And when you stop, you found the successor(31)

We found the replacement, but we still don't know how to remove the node and put its successor in its place.

Well, the trick is to do it in reverse order. We don't need to first remove and then replace. We can replace first and afterwards remove. Remember, the nodes themselves are just containers. What's inside them is what matters. So what we can do is to replace the value inside the node we want to remove with the value from the successor. So now we have the replacement in place (31)

but the old successor node is still in the tree (31).What we can do to address that might again not be very intuitive. Instead of writing some logic to remove the node, which will be more or less a duplication of what we already have, we can just recursively call the remove method on the old successor, and that will take care of locating and removing the node. And that's it

Why Go Leftmost After Right?

✅ The in-order successor of a node is the smallest value in the right subtree.

✅ To find this smallest value, we first move right (to access larger values) and then go left as far as possible (to find the smallest among them)

Example 50 / \ 30 70 / \ 60 80

✅ If removing 50, we move right to 70 (its right child).

✅ Then, we go left until we reach the smallest node (60).

✅ 60 becomes the replacement for 50, ensuring BST rules remain valid.

We need a value that’s the next larger than the node being deleted — which, by definition, lives in the right subtree. The leftmost node in the right subtree is the smallest of the values larger than the node, making it the in-order successor.

Could We Use the Left Subtree Instead?

Yes, we technically can — by picking the in-order predecessor, which is the rightmost node in the left subtree. It’s just a different strategy, and both are valid. But most implementations prefer the successor because:

  • It’s often easier to find and delete (especially if it's a leaf or has just one child).
  • Traversal code is slightly more intuitive going leftward from the right child.

Time And Space Complexity¶

So this searching part we already established can be O(n) or O(logn).But for this case of removing a node with two children, we are doing more than that.We are searching for the successor and then we are recursively calling remove for the successor. So we do three times more searching than we did for the other two cases.Does this affect the complexities in any way? Well.Not really.It's true that we do the searching three times, so in theory we'll have three times O(logn) for the best case and three times O(n) for the worst case. But remember when assessing complexities we discard any constants. So three times O(n) is still an O(n) complexity. And the same applies for O(logn).As far as the space complexity goes, it will be O(1)

Real-Life Applications of Binary Search Trees (BSTs)¶

Binary Search Trees (BSTs) are widely used in systems where fast searching, insertion, and deletion of sorted data is critical. Their ordered structure makes them ideal for indexing, hierarchical data storage, and efficient lookups.

🔑 Key Applications¶

1. Databases & Indexing¶
  • Database Indexes: BSTs (often in the form of B-Trees or B+ Trees) maintain sorted indexes for quick search, insert, and delete operations.
  • Range Queries: Efficient retrieval of records within a range (e.g., customers with IDs between 1000 and 2000).
  • Primary & Secondary Indexes: Organize both primary keys and secondary attributes for faster query execution.
2. File Systems & Operating Systems¶
  • Directory Structures: Store hierarchical file paths for quick navigation.
  • Memory Management: Manage free memory blocks for efficient allocation and deallocation.
3. Compilers & Expression Parsing¶
  • Syntax Trees: Abstract Syntax Trees (ASTs) represent program code for parsing and optimization.
  • Symbol Tables: Store variable/function names in sorted order for fast lookup during compilation.
4. Networking & Distributed Systems¶
  • Routing Tables: Organize routing information for efficient packet forwarding.
  • Load Balancing: Maintain sorted lists of servers/resources for quick allocation.
5. Search Engines & Information Retrieval¶
  • Keyword Indexing: Store keywords in sorted order for fast document retrieval.
  • Autocomplete Features: BSTs (or variants like Tries + BSTs) suggest results in search bars.
6. Finance & E-commerce¶
  • Stock Market Systems: Maintain sorted lists of stock prices for quick lookups and range queries.
  • Product Catalogs: Organize products by price, rating, or category for efficient filtering.

⚠️ Trade-offs & Limitations¶

  • Unbalanced BSTs can degrade performance to O(n) (like a linked list).
  • Self-balancing BSTs (AVL, Red-Black Trees) are often used in practice to maintain efficiency.
  • Memory overhead: BSTs require extra pointers, which can be costly in memory-constrained systems.
